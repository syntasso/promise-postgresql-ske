# PostgreSQL with Syntasso Kratix Enterprise

This Promise is an enhanced version of the [Open Source PostgreSQL
Promise](https://github.com/syntasso/promise-postgresql/tree/main/internal)
using extended functionality from Syntasso Kratix Enterprise (SKE).

In this example Promise we show:

- Best practices in Observability for Enterprise Environments
- How to use the [SKE Health
  Agent](https://docs.kratix.io/ske/installing-ske/ske-health-agent) to report
  Promise status back to SKE
- How to enable the Promise to be used by the Backstage and Cortex integrations

To install:
```
kubectl apply -f https://raw.githubusercontent.com/syntasso/ske-promise-examples/postgresql-example/promise.yaml
```

To make a resource request (small by default):
```
kubectl apply -f https://raw.githubusercontent.com/syntasso/ske-promise-examples/postgresql-example/resource-request.yaml
```

## Operational Lifecycle

This Promise emits useful Kubernetes events as it moves from installation to a
ready-to-use service. The important point is that events are raised at two
levels:

- Promise events for `promise/ske-postgresql`
- Resource events for `ske-postgresql/example`

You will also see supporting events from jobs, pods, `work`, and
`workplacement` objects in `kratix-platform-system`. Those are the execution
trail for the workflows.

To inspect the lifecycle end-to-end:

```bash
kubectl get events -A --sort-by=.lastTimestamp | rg 'ske-postgresql|backstage|workplacement|PipelineStarted|WorksSucceeded|ReconcileSucceeded'
```

### Promise install events

When the Promise itself is installed, these events are the main milestones:

| Event reason | Object | What it means |
| --- | --- | --- |
| `Available` | `promise/ske-postgresql` | The Promise definition has been accepted by Kratix and is usable. |
| `PipelineStarted` | `promise/ske-postgresql` | A Promise-level configure workflow has started. In this Promise you should expect `promise-configure`, and if Backstage integration is enabled, `backstage-gen` as well. |
| `SuccessfulCreate` | `job/...` | Kratix created the job that will execute one of the workflows. |
| `Scheduled`, `Pulled`, `Created`, `Started` | `pod/...` | The workflow pod was scheduled and its containers started. These are normal execution events for the runner containers such as `reader`, `work-writer`, `status-writer`, and the workflow container itself. |
| `AllWorkplacementsScheduled` | `work/...` | Kratix scheduled all generated outputs to their target destinations. |
| `WorkplacementReconciled` | `work/...` | The destination-specific workload was reconciled. `operation: created` means new material was written. `operation: unchanged` means the desired output already matched what was there. |
| `WorkloadsWrittenToStateStore` | `workplacement/...` | The rendered workload was persisted for a destination such as `dev` or `backstage`. This is one of the strongest signals that the workflow produced output successfully. |
| `Completed` | `job/...` | The workflow job finished successfully. |
| `ConfigureWorkflowCompleted` | `promise/ske-postgresql` | All Promise configure workflows have finished. |
| `WorksSucceeded` | `promise/ske-postgresql` | All work objects generated by the Promise are ready. |
| `ReconcileSucceeded` | `promise/ske-postgresql` | The Promise reconciliation completed successfully. This is the final success signal for Promise installation. |

For this Promise, the two most meaningful Promise-level outputs are:

- `promise-configure` writes the Promise workload for the selected destination, typically `dev`
- `backstage-gen` writes the catalog metadata for the `backstage` destination

### Resource request events

After applying [resource-request.yaml](./resource-request.yaml), you should see
the same execution pattern for the resource instance `ske-postgresql/example`:

| Event reason | Object | What it means |
| --- | --- | --- |
| `PipelineStarted` | `ske-postgresql/example` | Resource configure has started. For this Promise the workflow is `instance-configure`. |
| `SuccessfulCreate` | `job/...` | Kratix created the job that will reconcile the resource request. |
| `Scheduled`, `Pulled`, `Created`, `Started` | `pod/...` | The resource workflow pod and its containers are running. |
| `WorkplacementReconciled` | `work/...` | The resource output was reconciled for the destination. `created` usually appears on the first run; `unchanged` is expected on later reconciliations if nothing drifted. |
| `WorkloadsWrittenToStateStore` | `workplacement/...` | The rendered PostgreSQL workload was written to the state store for the target destination, for example `dev`. |
| `Completed` | `job/...` | The resource workflow job finished successfully. |
| `WorksSucceeded` | `ske-postgresql/example` | All work generated by the resource request is ready. |
| `ReconcileSucceeded` | `ske-postgresql/example` | The resource instance reconciled successfully. This is the final success signal for the resource request. |

### How to read repeated events

Some events will appear multiple times. That is normal.

- Repeated `WorkplacementReconciled` with `operation: unchanged` means Kratix rechecked desired state and found no drift.
- Repeated `AllWorkplacementsScheduled` means the scheduler is confirming the same desired placement.
- Container lifecycle events such as `Pulled`, `Created`, and `Started` are emitted once per container in the workflow pod, not once per workflow.

### What good looks like

A healthy Promise install ends with these Promise events:

- `Available`
- `ConfigureWorkflowCompleted`
- `WorksSucceeded`
- `ReconcileSucceeded`

A healthy resource request ends with these resource events:

- `PipelineStarted: instance-configure`
- `Completed`
- `WorksSucceeded`
- `ReconcileSucceeded`

If you see `WorkloadsWrittenToStateStore` for `dev` and `backstage`, the
workflows produced the expected outputs for both the platform destination and
the Backstage integration.
